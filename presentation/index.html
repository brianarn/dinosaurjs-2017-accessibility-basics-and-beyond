Americans with Disabilities Act

  • 1990: Americans with Disabilities act was enacted, prohibiting discrimination by places of public accomodation
  • 2006: NFB sued Target for their inaccessible website, citing it as a place of public accomodation
  • 2008: Settlement reached, $6mil settlement fund was established

  • USA: Americans with Disabilities Act of 1990

Principles & Guidelines

POUR

WCAG 2.0

Principles

Principles: POUR

  • Perceivable
  • Operable
  • Understandable
  • Robust

Perceivable

Can everyone consume this content?

Can everyone consume your content?

Examples:

  • Text alternatives for non-text content
  • Captions, other alternatives for multimedia content
  • Visible to users of varying levels of sight
  • Functions well with varying ATs

Operable

Can everyone use this content or application?

Can people use your stuff?

Examples:

  • Keyboard navigation (tab, spacebar, enter, etc)
  • Touch interfaces

Understandable

Does this even make sense?

  • Is the content readable? Free of typos?
  • Is the content succinct? Long-form content can be fatiguing to users of ATs.

Robust

Can it be used in a broad variety of environments?

Write once, accessible everywhere

The idea here is that it works well in a broad variety of environments. Different browsers, ATs, etc.

WCAG in Depth

Perceivable

1.1: Text Alternatives

  • 1.1.1: Non-text Content (A)

  • 1.1.1: All non-text content has text alternative, except for the following:

    • Controls
    • Input
    • Time-based media

1.2: Time-based media

  • 1.2.2: Caption (Prerecorded) (A)
  • 1.2.4: Captions (Live) (AA)
  • 1.2.6: Sign Language (Prerecorded) (AAA)
Berkeley removed 20k videos due to noncompliance with 1.2.2
  • 1.2.2: Captions provided for all prerecorded content, except when there is a clearly labeled alternative

  • 1.2.4: Captions provided for live audio content

    Note: Berkeley removal of 20k videos

  • 1.2.6: Sign language provided for all prerecorded audio content

1.3: Adaptable

  • 1.3.1: Info and Relationships (A)
  • 1.3.2: Meaningful Sequence (A)
  • 1.3.3: Sensory Characteristics (A)
  • 1.3.1: Information, structure, and relationships conveyed through presentation can be programmatically determined or are available in text. Ex: Form with required fields, tables with clear headers, etc
  • 1.3.2: When the sequence in which content is presented affects its meaning, a correct reading sequence can be programmatically determined. Ex: Content is in reading order in DOM
  • 1.3.3: Instructions provided for understanding and operating content do not rely solely on sensory characteristics of components such as shape, size, visual location, orientation, or sound.

1.4: Distinguishable

  • 1.4.1: Use of Color (A)
  • 1.4.3: Contrast (Minimum) (AA)
  • 1.4.4: Resize Text (AA)
  • 1.4.6: Contrast (Enhanced) (AAA)
  • 1.4.9: Images of Text (No Exception) (AAA)
Check out Success Criterion 1.4.8 for an example of a heavy AAA requirement
  • 1.4.1: Color is not the only visual means of conveying information.
  • 1.4.3: Contrast ratio of at least 4.5:1 except large text (3:1) or incidental/logotype
  • 1.4.4: Text can be resized up to 200% w/o loss of content or function
  • 1.4.6: Contrast must be at least 7:1, except large text (4.5:1) or incidental/logotype
  • 1.4.9: Images of text only used for pure decoration or where particular presentation is essential

Operable

2.1: Keyboard Operable

  • 2.1.1: Keyboard (A)
  • 2.1.2: No Keyboard Trap (A)
  • 2.1.1: All functionality of the content is operable through a keyboard interface without requiring specific timings for individual keystrokes, except where the underlying function requires input that depends on the path of the user's movement and not just the endpoints
  • 2.1.2: If keyboard focus can be moved to a component of the page using a keyboard interface, then focus can be moved away from that component using only a keyboard interface, and, if it requires more than unmodified arrow or tab keys or other standard exit methods, the user is advised of the method for moving focus away.

2.2: Enough Time

  • 2.2.1 Timing Adjustable (A)
  • 2.2.2 Pause, Stop, Hide (A)
2.2.1: Where time limits matter, either the user can turn off, adjust, or extend the time limit, or there is a valid exception for the time limit 2.2.2: For things that move, blink, scroll or auto-update, users should be able to pause, stop, or hide it, effectively

2.3: Seizures

  • 2.3.1 Three Flashes or Below Threshold (A)
  • 2.3.1: Web pages do not contain anything that flahes more than three times in any one second period, or the flash is below the general flash and red flash thresholds

Also, 2.3.2 is a strict adherence to nothing that flahes more than three times in any one second, period.

2.4: Navigable

  • 2.4.1: Bypass Blocks (A)
  • 2.4.2: Page Titled (A)
  • 2.4.3: Focus Order (A)
  • 2.4.4: Link Purpose (In Context) (A)
  • 2.4.7: Focus Visible (AA)
  • 2.4.1: A mechanism is available to bypass blocks of content that are repeated on multiple web pages
  • 2.4.2: Web pages have titles that describe topic or purpose
  • 2.4.3: Focusable components receive focus in an order that preserves meaning and operability
  • 2.4.4: The purpose of links can be determined from the link text alone, or the link txt with its programmatically determined link context, except where the purpose would be ambiguous in general
  • 2.4.7: Any keyboard operable user interface has a mdoe of operation where the keyboard focus is visible

Understandable

3.1: Readable

  • 3.1.1 Language of Page (A)
  • 3.1.2 Language of Parts (AA)
  • 3.1.3 Unusual Words (AAA)
  • 3.1.1: Default human language of the page can be programmatically determined
  • 3.1.2: The human language of passages or phrases can be programmatically determined, except for proper names, technical terms, words of indeterminate language, and words or phrases that have become part of the vernacular of the immediately surrounding text
  • 3.1.3: A mechanism is available for identifying specific definitions of words or phrases used in an unusual or restricted way, including idioms and jargon

3.2: Predictable

  • 3.2.1 On Focus (A)
  • 3.2.2 On Input (A)
  • 3.2.3 Consistent Navigation (AA)
  • 3.2.1: When any component receives focus, it does not initiate a change of context.
  • 3.2.2: Changing the setting of any user interface component does not automatically cause a change of context unless the user has been advised of the behavior before using the component.
  • 3.2.3: Navigational mechanisms that are repeated on multiple Web pages within a set of Web pages occur in the same relative order each time they are repeated, unless a change is initiated by the user.

3.3: Input Assistance

  • 3.3.1 Error Identification (A)
  • 3.3.2 Labels or Instructions (A)
  • 3.3.3 Error Suggestion (AA)
  • 3.3.4 Error Prevention (Legal, Financial, Data) (AA)
  • 3.3.1: If an input error is automatically detected, the item that is in error is identified and the error is described to the user in text.
  • 3.3.2: Labels or instructions are provided when content requires user input.
  • 3.3.3: If an input error is automatically detected and suggestions for correction are known, then the suggestions are provided to the user, unless it would jeopardize the security or purpose of the content.
  • 3.3.4: Web pages that cause legal commitments or financial transactions, need at least one of: reversible, checked for validity, confirmed

Robust

4.1: Compatible

  • 4.1.1 Parsing (A)
  • 4.1.2 Name, Role, Value (A)
  • 4.1.1: In content implemented using markup languages, elements have complete start and end tags, elements are nested according to their specifications, elements do not contain duplicate attributes, and any IDs are unique, except where the specifications allow these features.
  • 4.1.2: For all user interface components (including but not limited to: form elements, links and components generated by scripts), the name and role can be programmatically determined; states, properties, and values that can be set by the user can be programmatically set; and notification of changes to these items is available to user agents, including assistive technologies.

Applications?

ARIA

Accessible Rich Internet Applications

Three aspects of use

  • WAI-ARIA Roles
  • WAI-ARIA States and Properties
  • Managing Focus

Roles

Provides information to assistive technologies about how to handle elements

Most common: Widget, Landmark

States & Properties

Provides additional context and information to assistive technologies about elements

Ex: Providing labels, additional descriptions, checked/visibility states

Managing Focus

Ensuring that focus adheres to POUR through dynamic applications

Ex: Dialogs should trap tab key, but also support closure through Esc key
